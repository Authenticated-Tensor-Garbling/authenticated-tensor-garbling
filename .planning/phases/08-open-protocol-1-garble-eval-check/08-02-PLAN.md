<!-- Reference: 5_online.tex Construction 3, lines 201-210 -->

c_gamma per gate (D-09):
  c_gamma = (L_alpha XOR l_alpha) ⊗ (L_beta XOR l_beta) XOR (L_gamma XOR l_gamma)
          = v_alpha ⊗ v_beta XOR v_gamma                  [equals 0 for honest parties]

In Plan 03, the test harness assembles `c_gamma_shares: Vec<AuthBitShare>` by:
  1. For each gate (i, j), expanding the AND into XOR of selected D_ev-authenticated
     shares of l_alpha[i], l_beta[j], l_gamma*[(i,j)] (= correlated_auth_bit_shares[j*n+i]),
     l_gamma[(i,j)] (= gamma_auth_bit_shares[j*n+i]) plus public-bit MAC contributions.
  2. Pre-XORing the gen-side and eval-side D_ev-shares pairwise so that each entry's
     `value` is the reconstructed bit and the `key`/`mac`/`value` triple satisfies
     the standard IT-MAC invariant under delta_ev.

`check_zero` then loops over the resulting slice and:
  - Returns false on first share with `value == true` (reconstructed bit ≠ 0 → abort).
  - Returns false on first share with `mac != key.auth(value, delta_ev)` (tampered MAC → abort).
  - Returns true if all shares pass both checks.

This is recommendation (a) from 08-RESEARCH.md Open Question 1 — the simplest mechanic
consistent with D-07's fixed signature and D-08's "thin primitive" framing.

    //! Online phase primitives that span both garbler and evaluator views.
    //!
    //! Currently hosts `check_zero()` only. `open()` (ONL-01) and its wrong-delta
    //! negative test (ONL-02) are deferred to a later phase per Phase 8 CONTEXT.md
    //! D-01 — they will live in this module once the message-passing design is
    //! settled.

    use crate::sharing::AuthBitShare;
    use crate::delta::Delta;

    /// Verifies that the per-gate consistency-check vector reconstructs to zero
    /// AND that its IT-MAC under `delta_ev` is valid.
    ///
    /// `c_gamma_shares` is the caller-assembled vector of D_ev-authenticated
    /// shares of `c_gamma`, where (per Construction 3 in 5_online.tex line 206):
    ///
    ///   c_gamma = (L_alpha XOR l_alpha) ⊗ (L_beta XOR l_beta)
    ///                                  XOR (L_gamma XOR l_gamma)
    ///           = v_alpha ⊗ v_beta XOR v_gamma   [= 0 for honest parties]
    ///
    /// # Caller contract (per CONTEXT.md D-08)
    ///
    /// `check_zero` is a thin primitive — it does NOT know about the protocol
    /// structs or the c_gamma assembly. The caller MUST:
    ///   1. Compute `c_gamma_shares` as the linear combination of D_ev-shares
    ///      (l_alpha, l_beta, l_gamma, l_gamma\*) per the formula above.
    ///   2. Pre-XOR the garbler-side and evaluator-side shares pairwise (using
    ///      the `+` operator on `AuthBitShare`, which is field-wise XOR — see
    ///      src/sharing.rs lines 66-115) so that for each entry:
    ///        - `share.value` is the reconstructed c_gamma bit (gen.value XOR ev.value)
    ///        - `share.key`   = gen.key XOR ev.key
    ///        - `share.mac`   = gen.mac XOR ev.mac
    ///      The combined share then satisfies the standard IT-MAC invariant
    ///      `mac == key XOR value * delta_ev`, which is GF(2)-linear over
    ///      individually correct shares.
    ///
    /// Calling `share.verify(&delta_ev)` directly on a RAW (un-XORed) cross-party
    /// share will panic with "MAC mismatch in share" even on correctly-formed
    /// shares (the key and MAC fields come from different bCOT directions and
    /// commit under different deltas — see src/auth_tensor_pre.rs lines 305-336
    /// for the full explanation). The pre-XOR step is what makes this primitive
    /// safe to call.
    ///
    /// # Returns
    ///
    /// - `true`  ("pass" / "do not abort") if every share has `value == false`
    ///   AND `mac == key.auth(value, delta_ev)`.
    /// - `false` ("abort") on the first share that fails either check.
    ///
    /// Empty slice returns `true` (vacuously zero).
    pub fn check_zero(c_gamma_shares: &[AuthBitShare], delta_ev: &Delta) -> bool {
        for share in c_gamma_shares {
            // (1) Reconstructed bit must be 0. The caller pre-XORed the two
            // parties' shares, so `share.value` IS the reconstructed bit.
            if share.value {
                return false;
            }
            // (2) IT-MAC invariant under delta_ev.
            // mac == key XOR (value * delta_ev) === key.auth(value, delta_ev).
            // When value is false (the only path that reaches here) this
            // simplifies to `mac == key` (XOR with zero block).
            let want = share.key.auth(share.value, delta_ev);
            if share.mac != want {
                return false;
            }
        }
        true
    }

    mod tests {
        use super::*;

        }

                "any failing share in the slice must cause check_zero to abort");

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Caller's `c_gamma_shares` ↔ `check_zero` | Caller is responsible for pre-XORing gen and eval shares pairwise. A naive caller passing un-XORed cross-party shares would corrupt the IT-MAC check and either pass or fail incorrectly. |
| `check_zero` return value ↔ downstream control flow | Returning `false` means "abort"; the consumer in Plan 03 must propagate this signal correctly (e.g., `assert!(check_zero(...))` or test-side `assert!(!check_zero(...))`). |

Pure in-memory Rust library; threats are correctness-level.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-07 | Tampering (correctness — caller misuse) | `check_zero` consumer passing un-XORed cross-party shares | mitigate | Doc comment explicitly documents the caller-pre-XOR contract, references the `verify_cross_party` precedent at src/auth_tensor_pre.rs:305-336, and warns that calling `share.verify(delta)` directly on raw cross-party shares would panic. Plan 03's positive end-to-end test exercises the correct path. |
| T-08-08 | Information Disclosure (silent pass on tampered MAC) | `check_zero` MAC check | mitigate | The MAC equality check `share.mac != want` runs on EVERY share, not just the value-bit check, so a tampered MAC paired with `value: false` cannot silently pass. `test_check_zero_fails_on_invalid_mac` directly proves this. |
