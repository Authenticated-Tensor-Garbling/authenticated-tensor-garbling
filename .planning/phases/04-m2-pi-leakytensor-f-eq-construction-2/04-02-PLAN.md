### IMPORTANT paper-notation audit (Q2 from RESEARCH.md — RESOLVED here)

Paper notation `itmac{x_B}{Δ_A}` = "B's bit x_B authenticated under A's Δ_A" ⇒ the verifier holds Δ_A. In `transfer_a_to_b`, A is the sender holding `delta_b` as correlation key (see `src/bcot.rs:57-59`); `receiver_macs[i] = K[0] ⊕ choice[i]·delta_b`, i.e., authenticated under Δ_B NOT Δ_A.

**Resolution for this plan:** use the SAME cross-party layout that the pre-rewrite `leaky_tensor_pre.rs` uses (and that `verify_cross_party` is built to validate). Specifically, when the paper formula at line 222 says `tensor_garbler(n, m, Δ_A, key(x_B@A), C_A)`, map to:
- `Δ_A` ← `self.bcot.delta_a`
- `key(x_B@A)` ← `cot_x_a_to_b.sender_keys` (A's sender keys from transfer_a_to_b on x_B; this matches the cross-party layout at src/leaky_tensor_pre.rs:60-67 doc and the pre-rewrite tests that DID pass)
- `C_A` ← the m-length Vec<Block> computed per Pattern 2 in PATTERNS.md

And `tensor_evaluator(n, m, G_1, mac(x_B@A), C_B)`:
- `mac(x_B@A)` ← `cot_x_a_to_b.receiver_macs` (B's macs; held by B as the evaluator)

(Same resolution the RESEARCH.md Example 3 "Cleaner" block reaches. The pre-Plan-1 code passed bCOT-layer tests with this convention; the 4 failing tests were field-name and struct-shape issues, not layout issues. Plan 3 tests will verify end-to-end correctness — if the convention is inverted the IT-MAC / product invariants will fail there and we revisit.)
