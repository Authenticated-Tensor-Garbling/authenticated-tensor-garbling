## Summary

This phase implements Pi_LeakyTensor (Construction 2, Appendix F), the F_eq ideal functionality, the Pi_aTensor bucketing combiner (Construction 3), and the real-protocol preprocessing pipeline. The core cryptographic logic is sound: the GGM-tree macro, cross-party IT-MAC assembly, masked-reveal steps (S_1, S_2, D), and F_eq check all implement the paper correctly. The authenticated tensor gate (gen/eval) and the full online phase are well-structured with good test coverage.

The review found one critical bug in `bucket_size_for` that causes a division-by-zero panic for the degenerate `n=1, m=1` case (and an integer underflow for `n=0` or `m=0`), three warnings (a wrong doc-comment on `TensorFpreEval.delta_b`, an integer-overflow risk in the tweak computation of `tensor_ops`, and a stale `assert_eq!(count, 1, …)` guard that will block future batch use), and four info items (dead benchmark functions, debug `println!` in test code, a leftover TODO comment, and a code-style nit).

## Critical Issues

### CR-01: Division by zero in `bucket_size_for` when `n == 1, m == 1`

**File:** `src/auth_tensor_pre.rs:19-20`

**Issue:** For `ell = n * m = 1`, `ell.leading_zeros() == usize::BITS - 1 = 63`, so:

```
log2_ell = (usize::BITS - ell.leading_zeros() - 1) as usize
         = (64 - 63 - 1) = 0
```

The next line `SSP / log2_ell` then panics with a division-by-zero. The comment `// floor(log2(ell)) for ell >= 2` acknowledges the precondition but there is no enforcement — no `assert!` or early return guards against `ell < 2`. Similarly, when `ell == 0` (n=0 or m=0), `ell.leading_zeros() == 64` and the subtraction `64 - 64 - 1` underflows (wraps to `usize::MAX` in release mode, panics in debug). Neither situation is tested.

**Fix:**
```rust
pub fn bucket_size_for(n: usize, m: usize) -> usize {
    const SSP: usize = 40;
    let ell = n * m;
    assert!(ell >= 2, "bucket_size_for requires n*m >= 2 (got n={}, m={})", n, m);
    // floor(log2(ell)) for ell >= 2: leading_zeros gives bits above the MSB.
    let log2_ell = (usize::BITS - ell.leading_zeros() - 1) as usize;
    SSP / log2_ell + 1
}
```

Alternatively, if `ell == 1` is a valid input the formula should treat it as `log2(1) = 0` requiring infinite buckets, and the function should return a documented large constant or panic with a more descriptive message.

## Warnings

### WR-01: Doc comment on `TensorFpreEval.delta_b` incorrectly states `lsb() == 1`

**File:** `src/preprocessing.rs:43`

**Issue:** The field-level doc comment reads `"as_block().lsb() == 1 invariant"` for `delta_b`. However, in the real preprocessing protocol `delta_b` is generated by `Delta::random_b`, which explicitly clears the LSB to 0. The invariant `lsb(delta_a XOR delta_b) == 1` relies on `delta_a.lsb() == 1` AND `delta_b.lsb() == 0` (confirmed by the test `bcot::tests::test_delta_xor_lsb_is_one`). The wrong comment is a protocol-level misstatement — anyone reading the struct's contract to reason about MAC validity will be misled.

**Fix:**
```rust
/// Evaluator's (Party B) global correlation key.
/// `as_block().lsb() == 0` invariant (set by `Delta::random_b` in `IdealBCot::new`).
/// Paired with `delta_a` (lsb=1) so that `lsb(delta_a XOR delta_b) == 1`,
/// as required by Pi_LeakyTensor Construction 2's masked-reveal step.
pub delta_b: Delta,
```

### WR-02: Tweak value computation in `tensor_ops` can silently overflow for large `n`

**File:** `src/tensor_ops.rs:55-56`

**Issue:** The tweak values are computed as:

```rust
let tweak_even = Block::from(((i as u128) << 1) as u128);
let tweak_odd  = Block::from(((i as u128) << 1 | 1) as u128);
```

The cast `i as u128` is safe. However, if `n` grows beyond 127 (the loop runs `i` in `1..n`), `(i as u128) << 1` overflows `u128` for `i >= 128`, silently wrapping. In the current benchmarks `n` reaches at most 256 bits wide as a logical dimension, but `gen_populate_seeds_mem_optimized` is called with the GGM-tree depth which equals `chunking_factor` (up to 8 in practice), so this is not triggered today. A future caller with large `n` would silently produce tweak collisions, undermining domain separation. The same pattern appears in the evaluator path at lines 169-170.

**Fix:** Add a defensive bound check, or use a structured tweak construction that does not overflow:
```rust
assert!(n <= 127, "n must be <= 127 to avoid tweak overflow (i << 1 in u128)");
// or, better:
let tweak_even = Block::from(((i as u128) << 1));
let tweak_odd  = Block::from(((i as u128) << 1) | 1u128);
// and document the max-n constraint in the function's doc comment
```

### WR-03: `assert_eq!(count, 1, …)` in `run_preprocessing` will silently panic for any batch use

**File:** `src/preprocessing.rs:85`

**Issue:**

```rust
assert_eq!(count, 1, "Phase 1: only count=1 is supported; batch output requires Vec return");
```

The return type is `(TensorFpreGen, TensorFpreEval)` — a single pair. If a future caller passes `count > 1`, all `bucket_size * count` leaky triples are generated but only the first bucket is returned via `combine_leaky_triples`. The assertion prevents this silent data loss, which is correct. However, the generated triples for the extra buckets are computed and silently discarded before the assertion fires (the assertion is at the top, so actually it fires before generation — but this is still confusing). More importantly, the function signature promises a single output regardless of `count`, so the `count` parameter is misleading for callers.

**Fix:** Either remove the `count` parameter entirely and document that the function produces exactly one triple, or change the return type to `Vec<(TensorFpreGen, TensorFpreEval)>` for future batch use. At minimum, move the assertion to the very first line to avoid confusion:
```rust
pub fn run_preprocessing(n: usize, m: usize, chunking_factor: usize) -> (TensorFpreGen, TensorFpreEval) {
    // count removed — always produces one triple
    ...
}
```

## Info

### IN-01: `bench_full_protocol_garbling` and `bench_full_protocol_with_networking` are defined but never registered

**File:** `benches/benchmarks.rs:87-177`

**Issue:** Both `bench_full_protocol_garbling` (line 87) and `bench_full_protocol_with_networking` (line 118) are fully-implemented benchmark functions but are absent from the `criterion_group!(benches, …)` macro at line 603. Criterion will never invoke them. Similarly, `_setup_semihonest_gen` and `_setup_semihonest_eval` (lines 47-68) are prefixed with `_` to suppress unused-function warnings, indicating they are vestigial.

**Fix:** Either add the missing benchmarks to `criterion_group!` if they are still needed, or remove them and their setup helpers to avoid dead code accumulating.

### IN-02: Debug `println!` statements left in integration test in `src/lib.rs`

**File:** `src/lib.rs:326-329, 378`

**Issue:** The `test_auth_tensor_product` integration test contains several `println!` calls that print intermediate protocol state:

```rust
println!("gen_chunk_levels: {:?}", gen_chunk_levels.len());
println!("gen_chunk_levels[0] (each hold 2 blocks): {:?}", gen_chunk_levels[0].len());
println!("gen_chunk_cts: {:?}", gen_chunk_cts.len());
println!("gen_chunk_cts[0] (ecah hold one block): {:?}", gen_chunk_cts[0].len());
```

These are diagnostic artifacts from development (note the typo `"ecah"` at line 329). They add noise to `cargo test` output without asserting correctness.

**Fix:** Remove the `println!` calls, or if the information is useful for debugging, guard them with `eprintln!` and a `#[cfg(debug_assertions)]` gate.

### IN-04: `_setup_semihonest_*` helpers and commented-out `warm_up_time` lines in benchmarks
