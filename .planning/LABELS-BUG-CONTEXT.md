# Bug Context: Labels Incorrectly Embedded in Preprocessing

## Summary

Wire labels do not belong in the preprocessing layer. They were incorrectly
threaded through the leaky triple, Fpre structs, and into the online-phase
structs. This contaminates the preprocessing/online separation and makes triple
generation appear input-dependent.

---

## Root Cause

`LeakyTensorPre::generate()` originally accepted `x_clear: usize, y_clear:
usize` and used them in Step 4 to compute input-dependent wire labels:

```rust
let masked_bit = (((x_clear >> i) & 1) != 0) ^ alpha_bits[i];
let label_b = if masked_bit { label_0 ^ delta_a } else { label_0 };
eval_alpha_labels.push(label_b);
```

This is wrong on two levels:

1. **Triple generation must be input-independent.** Bucketing (Pi_aTensor,
   Construction 3) assembles triples before any input is known. Passing
   `x_clear`/`y_clear` into `generate()` violates this invariant. The
   workaround in `run_preprocessing` was to call `generate(0, 0)`, which
   silently produced labels for a fixed dummy input rather than the real one.

2. **Labels do not belong in a preprocessing triple at all.** The triple's
   responsibility is to output authenticated shares of the random masks (alpha,
   beta, alpha∧beta, gamma). Wire labels are a garbling artifact — they are
   generated by the garbler at garble time and do not flow from preprocessing.

---

## Affected Files and Their Role in the Bug

| File | Issue |
|---|---|
| `src/leaky_tensor_pre.rs` | `LeakyTriple` held `gen_alpha_labels`, `gen_beta_labels`, `eval_alpha_labels`, `eval_beta_labels`. `generate()` accepted `x_clear`/`y_clear`. Step 4 computed input-dependent eval labels. |
| `src/auth_tensor_pre.rs` | `combine_leaky_triples` forwarded `t0.eval_alpha_labels` / `t0.eval_beta_labels` into `TensorFpreEval`, propagating the incorrect labels downstream. |
| `src/auth_tensor_fpre.rs` | `run_preprocessing` called `ltp.generate(0, 0)` as a workaround, silently binding labels to the dummy input `0` instead of the real masked value. |
| `src/auth_tensor_fpre.rs` | `TensorFpreGen` and `TensorFpreEval` carry `alpha_labels`/`beta_labels` fields that originate from the triple. These fields are only meaningful for the ideal `TensorFpre` path. |
| `src/auth_tensor_gen.rs` | `AuthTensorGen.x_labels` / `y_labels` are populated from `fpre_gen.alpha_labels`. The garbler should generate fresh labels itself. |
| `src/auth_tensor_eval.rs` | `AuthTensorEval.x_labels` / `y_labels` are populated from `fpre_eval.alpha_labels`. The evaluator should receive labels from the garbler in the online phase (after `d = x XOR alpha` is revealed), not from preprocessing. |

---

## Partial Fix Already Applied (this session)

- Removed `x_clear`/`y_clear` from `LeakyTensorPre::generate()`.
- Removed `eval_alpha_labels` / `eval_beta_labels` from `LeakyTriple`.
- Step 4 now generates `label_0` (LSB=0) stored only in `gen_alpha_labels` /
  `gen_beta_labels`.
- `combine_leaky_triples` now copies gen labels to both gen and eval sides of
  `TensorFpre{Gen,Eval}`.
- `run_preprocessing` updated to `ltp.generate()` (no args).
- All tests pass (45/45).

**This partial fix is still wrong:** `gen_alpha_labels`/`gen_beta_labels`
should not exist in `LeakyTriple` at all. Labels remain in the struct and
continue to propagate through `TensorFpreGen`/`TensorFpreEval` into the online
phase structs.

---

## Correct Target Architecture

### Preprocessing (triple / Fpre)
- Outputs only: `alpha` shares, `beta` shares, `alpha∧beta` shares, `gamma` shares.
- No labels.
- `TensorFpreGen` and `TensorFpreEval` should drop `alpha_labels`/`beta_labels`
  for the real-protocol path (they can stay for the ideal `TensorFpre` path if
  that path is kept).

### Garbling (online — garbler side)
- `AuthTensorGen` generates fresh `label_0[i]` per wire at construction or
  garble time (independent of preprocessing).
- `x_labels[i]` = random block with LSB=0, generated by the garbler.

### Evaluation (online — evaluator side)
- `AuthTensorEval.x_labels[i]` is NOT set during preprocessing.
- After the garbler broadcasts `d = x XOR alpha`, the garbler sends the
  evaluator the wire label `label_0[i] XOR (d[i] * delta_a)`.
- `AuthTensorEval` receives and stores these labels as part of the online
  protocol message, not from Fpre.
- `get_first_inputs()` on the eval side should reflect this: either accept `d`
  as a parameter or be called after labels are set via a separate method.

---

## Invariant to Preserve

The ideal `TensorFpre` path (`generate_with_input_values`) sets up labels
correctly using the real input and is used in `test_auth_tensor_product`. This
path can continue to bundle labels in `TensorFpreGen`/`TensorFpreEval` if the
ideal Fpre is kept as a testing/reference tool. The real preprocessing path
(leaky triples + bucketing) must not touch labels.
